# NIC DMD — Delta Markov Duda

## Протокол сжатия для встроенных устройств и передачи через LoRa

[![Лицензия: GPL v3](https://img.shields.io/badge/Licence-GPLv3-red.svg)](https://www.gnu.org/licenses/gpl-3.0)

---

## Что такое DMD?

DMD — это многоплатформенный протокол сжатия для небольших пакетов данных с метеостанций, счётчиков электроэнергии, GPS-трекеров и других встроенных устройств. Он предназначен для передачи по технологиям с ограниченной полосой пропускания, таким как LoRa.

Протокол полностью работает на контроллере ATmega328 и не требует больших словарей или таблиц поиска в памяти. Каждый пакет сжимается независимо — адаптивным выбором лучшего метода из пяти кандидатов.

---

## Почему DMD?

Существующие библиотеки сжатия для встроенных устройств либо требуют сотни дополнительных байт RAM (Heatshrink), либо нуждаются в передаче таблицы Хаффмана вместе с данными. DMD выбирает другой путь — комбинирует несколько простых методов с эвристическим анализом и выбирает лучший результат для каждого пакета отдельно.

**Основные преимущества:**
- Фиксированная таблица Хаффмана только в ROM (32B), без накладных расходов на RAM
- Адаптивный выбор метода для каждого пакета — до 5 кандидатов
- Полностью детерминированная декомпрессия — без потери данных
- Максимальное расширение данных на 1 байт (заголовок) в худшем случае
- Реализация на Python и C (ATmega328 / Arduino)

---

## Когда DMD не выгоден

DMD предназначен для данных, которые меняются медленно и предсказуемо — показания датчиков, GPS-координаты, промышленная телеметрия. Если входные данные случайны, зашифрованы или уже сжаты, DMD добавит только 1 байт заголовка и отправит их как RAW. Это правильное поведение — без потерь при сжатии, без ухудшения качества.

---

## Совместимость

**Python:** 3.10 или новее (использует аннотации типов `bytes | None`).

**C:** C99 или новее. Протестировано с GCC на PC (Linux/Windows) и AVR-GCC для ATmega328. Без зависимостей от стандартной библиотеки, кроме `<string.h>`. Внутренние буферы размещаются с помощью C99 VLA в соответствии с фактической длиной пакета.

**Arduino:** Скопируйте `nic_dmd.c` и `nic_dmd.h` в папку проекта. Совместимо с Arduino IDE 1.8+ и 2.x (AVR-GCC поддерживает C99 VLA).

**Примечание для других компиляторов:** IAR, Keil и MSVC C++ не поддерживают VLA. Для этих инструментальных цепочек определите `-DDMD_PKT_MAX_BUILD=N` при компиляции (например, 32 или 64) и буферы будут фиксированного размера.

**Зависимости для fetch/benchmark:** `pip install requests`

**Длина пакета:** Техническое ограничение составляет 1B, но ниже 16B сжатие практически не выгодно — накладные расходы заголовка (1B) и состояние ANS (2B) поглощают большую часть потенциальной экономии. Рекомендуемый минимум — **16B**. Максимум — **255B**. Для передачи через LoRa практический лимит полезной нагрузки составляет 51–64B в зависимости от коэффициента распространения и региона. Наилучших результатов DMD достигает на данных, где соседние пакеты меняются медленно — обычно 16–64B сенсорной телеметрии.

## Валидация и целостность данных

В целях достижения максимальной производительности и абсолютной минимизации нагрузки на процессор библиотека не выполняет дополнительных проверок заголовков и валидации длины переданных данных.

Конструкция протокола строго предполагает, что проверка целостности (например, аппаратный CRC) и отбраковка повреждённых или пустых пакетов выполняется нижним транспортным уровнем или основной программой (как правило, самим радиомодулем, логикой сбора данных и т. д.). Пользователь библиотеки должен на уровне приложения обеспечить, чтобы в функции сжатия и декомпрессии поступали только структурно корректные данные. Такое делегирование ответственности позволяет достичь низких накладных расходов на память без напрасной траты тактов процессора.

## Потребление RAM

Буферы размещаются с помощью C99 VLA в соответствии с фактической длиной пакета `N`. Значения включают стек при сжатии и структуры энкодера/декодера.

```
================================================================================
  Длина пакета | Стек сжатия | dmd_encoder_t | dmd_decoder_t | Итого
--------------+-------------+---------------+---------------+---------
       16B    |     62B     |      18B      |     17B       |    80B
       32B    |     96B     |      34B      |     33B       |   130B
       64B    |    164B     |      66B      |     65B       |   230B
      128B    |    300B     |     130B      |    129B       |   430B
      255B    |    569B     |     257B      |    256B       |   826B
================================================================================
```

Пиковое потребление RAM при вызове `dmd_compress` = Стек сжатия + dmd_encoder_t.

При типичном использовании с LoRa (пакеты 16–64B) пик составляет **80–230B** — без проблем на ATmega328 (2KB RAM).

## Как это работает

```
+-------------------------------------------------------------------------------+
|                       НАЧАЛО: Входной пакет данных                            |
+-------------------------------------------------------------------------------+
                                    |
                                    v
+-------------------------------------------------------------------------------+
|  Шаг 1: Delta + ZigZag (keyframe пропускает этот шаг)                        |
|  Тест DELTA_1B / DELTA_2B / DELTA_FULL — выбрать минимальный счёт единичных  |
+-------------------------------------------------------------------------------+
                                    |
                                    v
+-------------------------------------------------------------------------------+
|  Шаг 2: Попробовать кандидатов сжатия (каждый с ограничением early exit)     |
|                                                                               |
|   (a) µANS     — только если zero_ratio >= 45%                               |
|   (b) Huffman  — nibble Huffman с фиксированной таблицей ROM, всегда         |
|   (c) FLAG     — битовая карта нулевых байтов, всегда                        |
|   (d) FLAG+HUF — FLAG удаляет нули, Huffman сжимает остаток                  |
+-------------------------------------------------------------------------------+
                                    |
                                    v
+-------------------------------------------------------------------------------+
|  Шаг 3: Выбрать наименьший результат                                         |
|  Если ничего не помогает → RAW резерв (delta_type = NONE, исходные данные)   |
+-------------------------------------------------------------------------------+
                                    |
                                    v
+-------------------------------------------------------------------------------+
|  Шаг 4: Собрать заголовок (1B) + полезная нагрузка → передать                |
+-------------------------------------------------------------------------------+
```

### Заголовок (1 байт)

Каждый сжатый пакет начинается с одного байта заголовка:

```
MSB                    LSB
 7    6    5    4    3    2    1    0
[huf][ans][flg][dlt][dlt][smp][smp][smp]
```

```
=======================================================================
| Биты |        Значение                                              |
|------|--------------------------------------------------------------|
|   7  | nibble Huffman сжатие      1 = ВКЛ                         |
|   6  | µANS сжатие                1 = ВКЛ                         |
|   5  | Флагирование нулевых байт  1 = ВКЛ                         |
|  4-3 | Тип дельты: 00=нет, 01=1B, 10=2B, 11=FULL (big-int+carry)  |
|  2-0 | Номер образца (0–7)        0 = keyframe / start frame       |
=======================================================================

Комбинация бит 7 + бит 5 = FLAG+HUF (карта нулей + Huffman на ненулевых)
```

Если ни один метод не может сжать данные лучше, чем RAW, исходные данные отправляются с битами 3-7 заголовка, установленными в 0. Приёмник распознаёт RAW, поскольку заголовок не использует никаких настроек.

### Слои сжатия

**1. Delta — дифференциальный метод**

Сравнение двух последовательных пакетов. Там, где данные меняются медленно (температура, давление, GPS-координаты), вычитание даёт цепочки нулевых или очень малых значений. Протокол тестирует три типа дельты и выбирает тот, который даёт наилучший результат по эвристике (количество единичных битов).

Поддерживаемые типы:
- **1B delta** — байт за байтом (арифметика uint8_t)
- **2B delta** — по 16-битным словам big-endian (арифметика uint16_t)
- **FULL delta** — весь пакет как одно большое целое число с переносом через все байты. Выигрывает на счётчиках и GPS-координатах, где значения переполняют байтовые границы.

**2. ZigZag кодирование**

После применения дельты данные преобразуются с помощью ZigZag-кодирования. Отрицательные разности отображаются в малые нечётные числа, положительные — в малые чётные числа. В результате получаются данные с большим количеством нулевых битов, которые лучше реагируют на последующие методы сжатия.

ZigZag не применяется, если delta = нет (включая keyframe). Delta и ZigZag выполняются за один проход по данным.

**3. Флагирование нулевых байтов (FLAG)**

Каждый нулевой байт заменяется одним битом в битовой карте. Перед картой хранится длина пакета (1B). Ненулевые байты следуют в исходном порядке.

Пример для пакета 16B с 12 нулями:
```
Исходные: [0, 0, 5, 0, 0, 0, 3, 0, 0, 0, 0, 0, 7, 0, 0, 2]  (16B)
Payload:  [16][11011101 11110110][5, 3, 7, 2]
           1B длина + 2B карта + 4B ненулевые = 7B
Результат: 8B вместо 16B (1B заголовок + 7B payload)
```

**4. µANS сжатие**

Асимметричные системы счисления (ANS) работают на уровне битов с двумя весами: нулевой бит высоко вероятен (29/32), единичный менее вероятен (3/32). Для данных с преобладанием нулей после delta+ZigZag достигает значительного сжатия без таблицы.

ANS payload содержит длину данных (1B), состояние (2B — uint16_t) и закодированные байты. Запускается только если доля нулевых байтов >= 45% (эвристика). Энкодер и декодер имеют early exit — если результат превышает лимит, вычисление немедленно останавливается.

**5. Nibble Huffman (HUF)**

Фиксированная таблица Хаффмана, обученная на комбинированных метео и GPS данных после delta+ZigZag. Кодирует каждый байт как два nibble-кода (hi и lo). Таблица хранится в ROM (32B PROGMEM на ATmega), без накладных расходов на RAM.

Максимальная длина кода — 6 бит, среднее значение ~3.2 бита на байт. Выигрывает особенно на IoT, промышленных и комплексных данных, где нули редки, но распределение nibble соответствует таблице.

**6. Комбинация FLAG+HUF**

FLAG сначала удаляет нулевые байты в битовую карту, затем Huffman сжимает оставшиеся ненулевые байты. Payload: `[1B длина][карта][1B valid bits HUF][HUF stream]`. Лучшее из обоих миров — детерминированное устранение нулей + энтропийное сжатие остатка.

**Keyframe и start frame**

Образец с номером 0 является keyframe. Поскольку предыдущий пакет для вычисления дельты отсутствует, дифференциальный метод и ZigZag пропускаются. Данные обрабатываются напрямую методами FLAG, HUF, FLAG+HUF или ANS. Keyframe происходит автоматически каждые 8 пакетов или после сброса устройства.

## Использование

### Python

```python
from nic_dmd import DmdEncoder, DmdDecoder

PKT_LEN = 16
enc = DmdEncoder(PKT_LEN)
dec = DmdDecoder(PKT_LEN)

data = bytes([0xFC, 0x18, 0x21, 0x34, 0x01, 0x81,
              0x04, 0xCE, 0x00, 0x00, 0xFC, 0x7C,
              0xFC, 0xA8, 0x00, 0x00])

compressed   = enc.compress(data)
decompressed = dec.decompress(compressed)

print(f"Сжато: {PKT_LEN}B → {len(compressed)}B")
assert decompressed == data
```

### C (ATmega328 / Arduino)

```c
#include "nic_dmd.h"

dmd_encoder_t enc;
dmd_decoder_t dec;

void setup() {
    dmd_encoder_init(&enc, 16);   // длина пакета — должна совпадать на обеих сторонах
    dmd_decoder_init(&dec, 16);
}

void loop() {
    uint8_t data[16]          = { /* данные датчика */ };
    uint8_t compressed[DMD_OUT_MAX];
    uint8_t decompressed[16];

    uint8_t comp_len = dmd_compress(&enc, data, compressed);
    lora.send(compressed, comp_len);

    // На стороне приёмника:
    dmd_decompress(&dec, compressed, comp_len, decompressed);
}
```

### Сборка без VLA (другие компиляторы)

Если ваш компилятор не поддерживает C99 VLA (IAR, Keil, MSVC C++), определите максимальную длину пакета при компиляции:

```
gcc -DDMD_PKT_MAX_BUILD=32 nic_dmd.c ...
```

Буферы будут скомпилированы с фиксированным размером 32B. Идеально для прошивок с одной фиксированной длиной пакета (типичный вариант использования Arduino).
