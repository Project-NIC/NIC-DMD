# NIC DMD — Delta Markov Duda

## Протокол сжатия для встраиваемых устройств и передачи через LoRa

[![Лицензия: GPL v3](https://img.shields.io/badge/Licence-GPLv3-red.svg)](https://www.gnu.org/licenses/gpl-3.0)

---

## Что такое DMD?

DMD — это многоплатформенный протокол сжатия для небольших пакетов данных с метеостанций, счётчиков электроэнергии, GPS-трекеров и других встраиваемых устройств. Он разработан для передачи по технологиям с ограниченной полосой пропускания, таким как LoRa.

Протокол полностью работает на контроллере ATmega328 и не требует больших словарей или таблиц поиска в памяти.

По этой причине после обширного тестирования он был оптимизирован даже за счёт несколько меньшей степени сжатия — снижение средней компрессии примерно на 3% (данные метеостанций) привело к снижению нагрузки на процессор примерно на 70%.

---

## Почему DMD?

Существующие библиотеки сжатия для встраиваемых устройств либо требуют сотни дополнительных байт ОЗУ (Heatshrink), либо нуждаются в передаче таблицы Хаффмана вместе с данными. DMD выбирает другой подход — комбинирует несколько простых методов с эвристическим анализом и выбирает наилучший результат для каждого пакета отдельно.

**Ключевые преимущества:**
- Никаких таблиц в ПЗУ или ОЗУ
- Адаптивный выбор метода для каждого пакета
- Полностью детерминированная декомпрессия — без потери данных
- Максимальное расширение данных на 1 байт (заголовок) в худшем случае
- Реализация на Python и C (ATmega328)

---

## Когда DMD не имеет смысла

DMD разработан для данных, которые меняются медленно и предсказуемо во времени — значения датчиков, GPS-координаты, промышленная телеметрия. Если входные данные случайны, зашифрованы или уже сжаты, DMD добавит только 1 байт заголовка и отправит их как RAW. Это корректное поведение — без сжатия с потерями, без ухудшения.

---

## Совместимость

**Python:** 3.10 или новее (использует аннотации типов `bytes | None`).

**C:** C99 или новее. Протестировано с GCC на ПК (Linux/Windows) и AVR-GCC для ATmega328. Нет зависимостей от стандартной библиотеки кроме `<string.h>`.

**Arduino:** Скопируйте `nic_dmd.c` и `nic_dmd.h` в папку проекта. Совместимо с Arduino IDE 1.8+ и 2.x.

**Зависимости для бенчмарка:** `pip install requests heatshrink2`

**Длина пакета:** Технический минимум — 1Б, но ниже 16Б сжатие практически не имеет смысла — накладные расходы заголовка (1Б) и состояния ANS (2Б) поглощают большую часть возможной экономии. Рекомендуемый минимум — **16Б**. Максимум — **255Б**. DMD достигает наилучших результатов на данных, где соседние пакеты меняются медленно — как правило, 16–64Б сенсорная телеметрия.

## Валидация и целостность данных

В целях достижения максимальной производительности и абсолютной минимизации нагрузки на процессор библиотека не выполняет никаких дополнительных проверок заголовков или валидации длины переданных данных.

Архитектура протокола строго предполагает, что проверку целостности (например, аппаратный CRC) и отбрасывание повреждённых или пустых пакетов осуществляет нижележащий транспортный уровень или основная программа (как правило, сам радиомодуль, логика сбора данных и т.д.). Пользователь библиотеки должен на уровне приложения гарантировать, что в функции сжатия и декомпрессии поступают только структурно корректные данные. Такое делегирование ответственности позволяет достичь низких накладных расходов по памяти без напрасной траты тактов процессора.

## Использование ОЗУ

Значения включают стек при сжатии и структуры энкодера/декодера.

```
===============================================================
  Длина пакета | Стек сжатия | dmd_encoder_t | dmd_decoder_t
--------------+-------------+---------------+--------------
      16Б     |     78Б     |      19Б      |     18Б
      32Б     |    126Б     |      35Б      |     34Б
      64Б     |    222Б     |      67Б      |     66Б
     128Б     |    414Б     |     131Б      |    130Б
     255Б     |    795Б     |     258Б      |    257Б
===============================================================
```

Пиковое ОЗУ при вызове `dmd_compress` = Стек сжатия + dmd_encoder_t.

При типичном использовании (пакеты 16–32Б) пик составляет **97–161Б** — без проблем на ATmega328 (2КБ ОЗУ).

## Как это работает

```
+---------------------------------------------------------------------------+
|                    НАЧАЛО: Входной пакет данных                           |
+---------------------------------------------------------------------------+
                             |                                     |
                             v                                     |
+--------------------------------------------------------+         |
|  Предобработка: Delta, ZigZag и сохранение состояния   |         |
+--------------------------------------------------------+         |
            |                              |                       |
            v                              v                       |
+-----------------------+      +------------------------+          |
|   Метод: µANS         |      |  Метод: Флагирование   |          |
| [ Попытка сжатия ]    |      | [ Оценка карты нулей ] |          |
+-----------------------+      +------------------------+          |
            |                              |                       |
            v                              v                       |
+-------------------------------------------------------------+    |
| Решение: результат (µANS или флагирование) меньше RAW?     |    |
+-------------------------------------------------------------+    |
            |                                          |           |
            v                                          v           v
+-----------------------+                 +---------------------------+
|  ДА: Сжато            |                 |  НЕТ: Отправить как RAW   |
| [ Сформировать заголовок]               | [ Заголовок RAW ]         |
+-----------------------+                 +---------------------------+
```

### Заголовок (1 байт)

Каждый сжатый пакет начинается с одного байта заголовка:

```
MSB                    LSB
 7    6    5    4    3    2    1    0
[rsv][ans][flg][dlt][dlt][smp][smp][smp]
```

```
=========================================================================
| Биты |        Значение                                                |
|------|----------------------------------------------------------------|
|   7  | Резерв (для будущего расширения)                               |
|   6  | Сжатие µANS                1 = ВКЛ                            |
|   5  | Флагирование нулевых байт  1 = ВКЛ                            |
|  4-3 | Тип дельты: 00=нет, 01=1Б, 10=2Б, 11=полный образец           |
|  2-0 | Номер образца (0–7)        0 = ключевой кадр / стартовый кадр |
=========================================================================
```

### Слои сжатия

**1. Дельта — разностный метод**

Сравнение двух последовательных пакетов. Там где данные меняются медленно (температура, давление, GPS-координаты), вычитание даёт цепочки нулевых или очень малых значений. Протокол проверяет четыре типа дельты и выбирает лучший результат по эвристике (количество единичных битов).

Поддерживаемые типы:
- **1Б дельта** — байт за байтом
- **2Б дельта** — по двухбайтовым группам (подходит для 16-битных датчиков)
- **Полный образец** — каждый байт минус тот же байт предыдущего пакета
- **Без дельты** — данные проходят без разностного кодирования

**2. Кодирование ZigZag**

После применения дельты данные преобразуются кодированием ZigZag. Отрицательные разности отображаются в малые нечётные числа, положительные — в малые чётные. Результат — данные с высоким числом нулевых битов, лучше реагирующие на последующие методы сжатия.

ZigZag не применяется при дельта = нет (включая ключевой кадр).

**3. Флагирование нулевых байт**

Каждый нулевой байт заменяется одним битом в карте. Перед картой сохраняется длина пакета (1Б), чтобы декодер знал длину данных без предварительной договорённости. Ненулевые байты следуют в исходном порядке.

Пример для пакета 16Б с 12 нулями:
```
Исходные: [0, 0, 5, 0, 0, 0, 3, 0, 0, 0, 0, 0, 7, 0, 0, 2]  (16Б)
Полезная нагрузка: [16][11011101 11110110][5, 3, 7, 2]
                    1Б длина + 2Б карта + 4Б ненулевые = 7Б
Результат: 8Б вместо 16Б (1Б заголовок + 7Б полезная нагрузка)
```

**4. Сжатие µANS**

Асимметричные числовые системы (ANS) работают на уровне битов с двумя весами: нулевой бит весьма вероятен (29/32), единичный — менее (3/32). Для данных с преобладанием нулей после дельты+ZigZag достигается значительное сжатие без таблицы.

Полезная нагрузка ANS содержит длину данных (1Б), состояние (2Б) и закодированные байты. Декодеру не нужно знать длину заранее.

ANS фактически сжимается и измеряется. Флагирование оценивается детерминированно. Выбирается более короткий из двух результатов. Если сжатие увеличило бы размер данных, отправляются исходные данные — максимум 1 байт накладных расходов.

**Ключевой кадр и стартовый кадр**

Образец с номером 0 является ключевым кадром. Поскольку предыдущего пакета для вычисления дельты не существует, разностный метод и ZigZag пропускаются. Данные обрабатываются непосредственно флагированием или µANS. Ключевой кадр возникает автоматически каждые 8 пакетов или после сброса устройства.

## Использование

### Python

```python
from nic_dmd import DmdEncoder, DmdDecoder

PKT_LEN = 16
enc = DmdEncoder(PKT_LEN)
dec = DmdDecoder(PKT_LEN)

data = bytes([0xFC, 0x18, 0x21, 0x34, 0x01, 0x81,
              0x04, 0xCE, 0x00, 0x00, 0xFC, 0x7C,
              0xFC, 0xA8, 0x00, 0x00])

compressed   = enc.compress(data)
decompressed = dec.decompress(compressed)

print(f"Сжато: {PKT_LEN}Б -> {len(compressed)}Б")
assert decompressed == data
```

### C (ATmega328 / Arduino)

```c
#include "nic_dmd.h"

dmd_encoder_t enc;
dmd_decoder_t dec;

void setup() {
    dmd_encoder_init(&enc, 16);   // длина пакета — должна совпадать с обеих сторон
    dmd_decoder_init(&dec, 16);
}

void loop() {
    uint8_t data[16]          = { /* данные датчиков */ };
    uint8_t compressed[DMD_OUT_MAX];
    uint8_t decompressed[16];

    uint8_t comp_len = dmd_compress(&enc, data, compressed);
    lora.send(compressed, comp_len);

    // На стороне приёмника:
    dmd_decompress(&dec, compressed, comp_len, decompressed);
}
```
